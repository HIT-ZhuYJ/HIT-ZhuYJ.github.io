Claude Code 的记忆机制到底强在哪

视频链接：https://www.bilibili.com/video/BV1z6SXBzEYh/?spm_id_from=333.1007.top_right_bar_window_history.content.click&vd_source=0ee7dc7b90ab469b25a0f6fb060f8ebf

一个判断：Claude Code 强的不只是模型，而是“记忆系统工程”

很多人讨论 Claude Code 时，第一反应是“模型能力强”。但如果从 Agent 工程的角度看，它真正拉开差距的地方，往往不只是模型本身，而是以下几个系统能力的配合：

  • 能不能把规则稳定地注入上下文
  • 能不能在长会话里尽量少丢信息
  • 能不能跨会话保留真正有价值的记忆
  • 能不能在多人协作和多代理并发下避免记忆冲突

换句话说，Claude Code 的优势很可能不在“记得更多”，而在“知道什么该记、什么时候记、怎么安全地取回来”。

1. 指令记忆：先解决“AI 应该如何工作”

第一层可以理解为“指令记忆”。它解决的不是“用户说过什么”，而是“这个 Agent 在当前环境中应该按什么规则做事”。

视频里提到的核心载体是 CLAUDE.md 及其相关文件族。按照我的理解，这类文件承担的是一种“项目级操作规范”角色，例如：

  • 团队共享规则
  • 个人私有规则
  • 项目内规则
  • 更细目录下的局部规则

如果实现得合理，这类规则通常会按目录层级被收集，然后由近到远决定优先级。更靠近当前工作目录的规则，往往应当覆盖更外层的通用规则。这个设计很像配置系统里的“就近覆盖原则”。

三种常见扩展机制

从工程上看，这类规则体系一般会需要三种扩展能力：

  1. include 递归引入
    用于把大规则拆成多个模块，避免一个文件无限膨胀。

  2. 条件规则
    例如某些规则只在特定目录、特定文件类型或特定任务下生效。

  3. 工具或目录的上下文附着规则
    当 Agent 访问某个工具、目录或子模块时，自动补充该局部上下文相关的规则。

2. 双轨注入：不是所有规则都该塞进系统提示词

这些规则是全部放到系统提示词里么？

一个更合理的理解是“分层注入”：

  • 相对稳定、平台内建、很少变化的行为约束，适合放进系统提示词
  • 经常变化、和项目强相关的规则，适合放进用户侧上下文

双轨注入指的是

  • 通道 A：注入项目级和用户级规则，例如 CLAUDE.md 文件族。注入用户提示词。
  • 通道 B：注入记忆系统自身的调用规范和稳定机制，注入系统提示词。便于缓存。

这样做的好处很明显：

  • 可变内容不污染系统级缓存
  • 稳定规则可以被更高效地复用
  • 项目规则更新时，不需要重写整套底层行为约束

从 Agent 工程角度说，这其实是在平衡两个目标：可更新性和可缓存性。

3. 短期记忆：核心不是“保存历史”，而是“保住任务状态”

很多人把短期记忆理解成“完整聊天记录”，但对代码 Agent 来说，仅仅保留历史的摘要并不够。因为当上下文不断变长时，真正贵的不是“对话文本”，而是中间过程中的状态信息。

我更愿意把这一层定义为：

短期记忆 = 当前会话里为了继续完成任务而必须保留的执行状态。

如果上下文越来越长，直接粗暴压缩会丢掉很多关键信息，比如：

  • 当前任务目标
  • 已经尝试过的方法
  • 失败原因
  • 关键文件和函数
  • 下一步待执行事项

因此，一个成熟的系统往往不会等上下文爆炸后再整体总结，而是会持续维护一份会话摘要 memory。

一份好的会话摘要应该包含什么

  • 会话标题或任务名称
  • 当前工作状态
  • 明确的任务规格
  • 关键文件、模块和函数
  • 已完成步骤与待办步骤
  • 错误、回滚和修正过程

为什么要强调消息边界完整性

比如：

  • 不能把一组问答切断在中间
  • 不能把一次工具调用拆坏
  • 不能让同一条消息的片段落在不同语义边界里

否则压缩后的上下文虽然更短，但会变得难以解释，甚至让模型产生错误归因。

4. 工作记忆：Agent 真正依赖的是“当前执行面板”

工作记忆是 Agent 在当前任务执行窗口中持续更新的操作面板。它通常包括：

  • 当前步骤进行到哪里
  • 下一步打算做什么
  • 哪些假设已经验证，哪些还只是猜测
  • 哪些分支方案被放弃了

5. 长期记忆：真正关键的是“记什么，不记什么”

长期记忆是这套系统最有价值的部分，但也最容易被误解。

很多人以为长期记忆就是“把所有对话都存下来”，但这其实是最低效、也最危险的做法。一个成熟的长期记忆系统，关键不在于存得多，而在于筛得准。

更适合长期保存的内容

  • 用户画像：角色、偏好、表达风格
  • 行为反馈：用户如何纠正过 Agent，什么样的行为是被明确鼓励的
  • 项目上下文：长期有效的约束、目标和背景
  • 外部引用：文档、知识源、外部系统的稳定入口
  • 重要时间信息：尽量写成绝对日期，避免“昨天”“下周”这类会过期的相对表述

不适合写入长期记忆的内容

  • 代码模式
  • 文件路径
  • Git 历史
  • 临时调试方案

原因不是这些信息不重要，而是它们本来就应该以代码仓库、文件系统和版本历史作为唯一真实来源。如果再额外存进记忆，很容易和真实代码状态发生冲突。

这也是 Agent 设计里很重要的一个原则：

记忆系统应该存“代码之外的信息”，而不是复制“代码本身已经表达的信息”。

6. MEMORY.md 与动态召回：长期记忆不能全量塞进上下文

一种合理的长期记忆组织方式通常是两层：

  • 第一层：索引文件，例如 MEMORY.md
  • 第二层：具体记忆条目

索引负责让系统知道“有哪些记忆可供选择”，具体条目负责保存细节。

为什么必须做检索和筛选

如果长期记忆越来越多，不可能每次都全量放进上下文。更合理的做法是：

  1. 先读取轻量级索引或每条记忆的摘要
  2. 再根据当前任务选择最相关的少量记忆
  3. 最后把这些记忆注入当前上下文

7. 自动记忆提取：异步总结比同步打断更合理

自动记忆提取这部分是 Agent 工程里非常实用的一招。

如果主代理一边执行任务，一边频繁自己总结和写长期记忆，会明显打断主链路。因此更好的方式通常是：

  • 主代理专注当前任务

  • 辅助代理异步回顾对话

  • 子代理异步总结

  • 主代理与写记忆流程互斥

  • 先读后写，分两阶段执行

  • 写记忆的代理权限收窄，只允许写记忆相关资源。

8. 休眠重塑

休眠重塑

一个合理的整合流程通常包括：

  1. 浏览记忆目录和索引
  2. 查找重复、冲突和过时内容
  3. 合并相近记忆
  4. 将相对日期改写为绝对日期
  5. 删除被合并或失效的旧记忆
  6. 修剪索引长度，避免索引本身再次失控

9. 并发安全：记忆系统最怕的不是丢，而是乱

对一个多代理系统来说，记忆写入最危险的问题往往不是“写失败”，而是：

  • 两个代理同时改同一份记忆
  • 一个代理在整理时，另一个代理还在追加
  • 读取方拿到的是中间态数据

所以，一个成熟实现通常需要某种并发保护，例如：

  • 锁文件
  • 两阶段写入
  • 原子替换
  • 冲突检测与退让

10. 记忆隔离：多 Agent 协作不能共享一切

ClaudeCode中允许创建多个agent，但他们的的记忆不共享。

11. 缓存与 Feature Flag：强系统一定有“开关”和“缓存”

有以下缓存层：

  • memory 文件解析结果缓存
  • 拼接后的上下文对象缓存
  • 系统提示词缓存

这样做的目的不是炫技，而是为了减少重复解析、降低延迟，并提高稳定性。

至于“无热重载”这件事，也很合理。因为如果底层缓存存在，那么某些配置文件即便被修改，也未必会在当前轮次立刻生效。

12. 我的总结

如果把 Claude Code 的记忆机制压缩成一句话，我会这样概括：

它不是在追求“把所有东西都记住”，而是在构建一套面向 Agent 执行过程的分层记忆基础设施。

这套基础设施至少同时解决了四个问题：

  • 长上下文下的信息保真
  • 跨会话的用户与项目连续性
  • 多代理协作时的状态同步与隔离
  • 记忆写入、召回、整理过程中的工程稳定性

这也是为什么我会觉得，Claude Code 强的不只是模型，而是它把“记忆”真正做成了一个系统，而不是一个临时补丁。
